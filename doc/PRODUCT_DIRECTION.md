# Linx 产品定位

最后更新：2026-03-25

## 产品定义

Linx 是一个面向小团队和朋友联机的桌面组网客户端。它的职责不是承载聊天，而是帮助用户更快完成以下动作：

1. 创建房间或加入房间
2. 自动建立虚拟网络
3. 查看成员是否在线、是否可达
4. 复制虚拟 IP 或启动游戏
5. 在失败时快速诊断原因

## 目标用户

- 想和朋友远程联机，但不想手动折腾虚拟局域网配置的玩家
- 需要稳定房间、成员状态、IP 分发和诊断能力的小团体
- 更关心“能不能连上”而不是“能不能聊天”的用户

## 核心价值

- 降低联机门槛：把网络配置收敛到房间流程里
- 降低失败成本：失败时给出可操作的原因和修复建议
- 降低沟通成本：用用户可理解的词描述网络状态

## 主流程

```text
打开客户端
-> 登录
-> 创建房间 / 加入房间
-> 自动建立虚拟网络
-> 查看成员状态与虚拟 IP
-> 复制 IP / 启动游戏
-> 如失败则进入诊断
```

## 产品原则

### 1. 首页是动作页，不是信息流

首页优先展示：

- 创建房间
- 加入房间
- 最近房间
- 最近联机伙伴
- 当前网络状态

不优先展示：

- 会话列表
- 消息流
- 社交动态

### 2. 房间是网络容器，不是聊天容器

房间页的第一优先级是：

- 房主是谁
- 谁在线
- 谁不可达
- 当前是直连还是中继
- 我的虚拟 IP 是什么
- 下一步该做什么

### 3. 诊断能力必须前置

组网工具的体验上限不是“功能多”，而是“出问题时能不能定位”。诊断入口必须在首页或房间页一级可见。

### 4. 文案必须去 IM 化

避免继续强化“聊天软件”心智。核心概念以联机、房间、可达、节点、服务节点、联机已就绪为主。

## 非目标

当前阶段不作为主线推进的内容：

- 聊天窗口
- 会话列表
- 已读未读
- 通知中心
- 用户主页
- 社交资料卡
- 复杂好友搜索

## 核心实体

建议后续统一围绕以下实体建模：

- `user`
- `device`
- `room`
- `roomMember`
- `peerConnection`
- `gameProfile`
- `diagnosisRecord`

## 当前代码与目标之间的差距

当前仓库仍保留较多 IM 结构：

- 前端存在 `chat`、`friend`、`group` 类型与 store
- 后端存在 `ChatController`、`GroupController`、`FriendshipController`
- 主页面 `front/src/views/pages/lobby/index.vue` 已接入房间主链路，但仍需继续拆分独立路由视图与状态中心

这不是问题本身，但后续新增能力应优先围绕 `room` 和 `connection` 建模，避免继续投资在 IM 语义上。
